The Problem

What is the N+1 query problem?

It happens when you load a list of N items, then make one additional query for each item to load related data. With a second level of nesting (clients → entities → deliverables), it becomes N×M+1.

1
clients
+
200
entities
+
1,000
deliverables
=
1,201 queries
per page load

This pattern often starts as clean-looking code — each layer fetches what it needs. It's only when you multiply by real data volumes that the cost becomes visible.

// Looks clean. Hides the problem. const clients = await getClients() // 1 query for (const client of clients) { client.entities = await getEntities(client.id) // +1 per client = 200 queries for (const entity of client.entities) { entity.deliverables = await getDeliverables(entity.id) // +1 per entity = 1,000 queries } }
Why is this so slow in practice? Even if each query takes only 2ms in the database, each one is a separate round-trip through the network, ORM, connection pool, and API layer. At 2 seconds per call with a concurrency limit of 6, 1,201 queries becomes 201 batches — over 6 minutes of wall-clock time before the user sees data.

How the waterfall loads data

The rendering problem compounds the query problem. Once data finally arrives, the browser must create and layout 10,000 DOM nodes — each with cells, text, badges, and event listeners. On a mid-range device this alone adds several seconds and makes scrolling janky.

The optimized approach addresses queries, rendering, and live updates as separate problems with separate solutions.

Where the Performance Comes From

Three independent optimizations

🔗
1. Query Consolidation
Move the join into the database where it belongs. SQL was designed to traverse relationships across tables in a single pass using indexes. Doing it in application code with a loop is reconstructing a JOIN, badly.
1,201 round-trips → 1 query
DB indexes on client_id, entity_id make the join O(n log n)
🪟
2. Virtual Rendering
The browser only needs to display what fits in the viewport — roughly 20 rows. Virtual scroll keeps a flat array of all rows in memory, but only creates DOM nodes for the visible slice. The scrollbar is simulated with a spacer div.
10,000 DOM nodes → ~20 DOM nodes
Constant memory regardless of dataset size
⚡
3. WebSocket Deltas
Instead of polling or reloading to get live status updates, a persistent WebSocket connection lets the server push only what changed. A single status change is a ~60-byte message, not a 10,000-row reload.
Full reload on change → ~60 byte delta
UI stays live without any polling overhead
The Query

What the flat query looks like

The entire data payload — all 10,000 deliverables with their client and entity context — is returned by a single SQL statement. The database uses foreign key indexes to execute this in milliseconds.

-- One query replaces 1,201 SELECT d.id, d.type, d.period, d.status, d.due_date, d.assigned_to, e.id AS entity_id, e.name AS entity_name, e.entity_type, c.id AS client_id, c.name AS client_name, c.industry, c.region FROM deliverables d JOIN entities e ON d.entity_id = e.id -- idx_del_entity JOIN clients c ON e.client_id = c.id -- idx_ent_client ORDER BY c.name, e.name, d.type

The result is a flat list of denormalized rows — each row carries its own client and entity context. The browser reassembles the tree client-side using a two-pass Map lookup, which is O(n) and takes about 1ms for 10,000 rows.

// Client-side tree build — O(n), one pass const clientMap = new Map(), entityMap = new Map() for (const row of flatRows) { if (!clientMap.has(row.client_id)) clientMap.set(row.client_id, { ...clientFields(row), entities: [] }) if (!entityMap.has(row.entity_id)) { const e = { ...entityFields(row), deliverables: [] } entityMap.set(row.entity_id, e) clientMap.get(row.client_id).entities.push(e) } entityMap.get(row.entity_id).deliverables.push(deliverableFields(row)) }
Sorting

How column sort triggers a new stream

Sorting in the flat view can't be done cheaply in the browser — the dataset is 100,000 rows and the browser only holds what has been streamed so far. Instead, clicking a column header fires a new server-sent event stream with the requested sort order, and the server re-runs the query with a different ORDER BY. The table repopulates as the first batch of rows arrives — exactly like the initial load.

Sort click → re-stream flow (flat view)
🌐

User clicks Due Date ↑ header → setSort('due_date')

⚙️
API Server
Validates sort column against a whitelist (prevents SQL injection)
Builds query with dynamic ORDER BY:
// Only these columns can be sorted — anything else falls back to default const SORT_COLS = { due_date: 'd.due_date', client_name: 'c.name', service_line: 'sl.name', assigned_to: 'ps.name', period: 'd.period', status: 'd.status', } const col = SORT_COLS[req.query.sort] // 'due_date' → 'd.due_date' const dir = req.query.dir === 'desc' ? 'DESC' : 'ASC' const order = col ? `${col} ${dir}, d.id` : 'c.name, e.name, d.type' // ↑ d.id tiebreaker ensures stable sort when values are equal
↓
🗄️
Database
Same 1 JOIN query — different ORDER BY d.due_date ASC, d.id
Index on deliverables(due_date) makes sorted reads efficient
↓
⚙️
API Server — Streaming
Uses a cursor (stmt.iterate()) to stream rows without loading all 100,000 into memory
Sends batches of 5,000 rows as SSE events — first batch arrives in ~100ms
↓
🌐
Browser
Receives each batch, appends rows to deliverableMap, re-renders visible slice
Table populates progressively — user sees sorted rows immediately, more load in background
Tree view sorts differently. In the grouped tree view (Client, Entity, or Client + Entity), all 100,000 deliverables are already in memory from the initial load. Clicking a column header re-sorts the in-memory array client-side and rebuilds the visible rows — no server round-trip at all. Sort + re-render takes ~2ms for 100,000 rows.

The distinction matters: flat view needs server-side sort because it can't hold the full dataset in sorted order client-side without re-querying. Tree view already has everything in memory, so it can sort for free. Both use the same column headers and sort state — the difference is invisible to the user.

Virtual Rendering

How virtual scroll works

A virtual list maintains two things: a flat array of all rows (in memory, not DOM), and a viewport that only renders the slice currently visible. As the user scrolls, rows are recycled — the same ~20 DOM nodes are repositioned and their content is swapped.

Virtual scroll mechanics
Memory
visibleRows[] — 1,200 items
Each item: plain JS object, no DOM
Filters/group changes rebuild this array
Expand/collapse changes this array
~300KB RAM for 10,000 rows
DOM
Spacer div: height = rows × 42px
Row container: transform: translateY
Only renders scroll window ± 8 rows
On scroll: recalculate slice, swap content
~20 DOM nodes regardless of dataset size
// Virtual render — called on every scroll event function renderRows() { const scrollTop = viewport.scrollTop const viewH = viewport.clientHeight const totalRows = visibleRows.length // Which rows are in the viewport? const startIdx = Math.max(0, Math.floor(scrollTop / ROW_H) - OVERSCAN) const endIdx = Math.min(totalRows, Math.ceil((scrollTop + viewH) / ROW_H) + OVERSCAN) spacer.style.height = (totalRows * ROW_H) + 'px' // fake total height rowContainer.style.transform = `translateY(${startIdx * ROW_H}px)` // position visible slice rowContainer.innerHTML = visibleRows .slice(startIdx, endIdx) .map(row => renderRow(row)) // only render what's visible .join('') }
