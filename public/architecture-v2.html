V2 Architecture

V2: PostgreSQL + Progressive Loading

V2 builds on the V1 proof-of-concept by replacing the local SQLite database with a real PostgreSQL instance, introducing partial/progressive data loading, server-side filtering, a collapsible tree view, and dynamic column selection.

Where V1 demonstrated that a single flat JOIN query is far faster than N+1 waterfall fetching, V2 applies that same principle at real scale — 100,000 rows seeded from the production engagement schema — and solves the problems that emerge when you can't (or shouldn't) load everything at once.

Loading Strategy

Progressive Loading

Rather than streaming all 100,000 rows on every page load, V2 uses a 1,000-row buffer strategy. The UI is usable within milliseconds of opening the page.

Progressive load flow
💻
Initial load
Request first 1,000 rows (no cursor). Tree and flat view build progressively as rows stream in. Top of list appears populated almost immediately.
↓
💻
Filter applied
Client instantly filters already-loaded rows. Result appears with zero network latency.
↓
💻
Threshold check
If visible rows < 1,000 AND server has more rows: auto-fetch next batch with server-side filters applied (WHERE clause pushed to DB).
↓
⚙️
Server fetch with filters
Server executes SELECT ... WHERE office = $1 AND tax_year = $2 AND id > $3 ORDER BY ... LIMIT 1000 — cursor-based, filtered.
↓
💻
Merge and re-filter
New rows merged into client store. Client filter re-applied. Threshold checked again. Process repeats until 1,000 visible rows or server exhausted.
💾
Sort change
Cursor resets. Server refetches from beginning with new ORDER BY. Client store cleared and rebuilt from new stream.
Pagination

Cursor-Based Pagination

Traditional OFFSET pagination scans and discards rows on every page request. At 50,000 rows in, the database does 50,000 units of work it throws away. Cursor (keyset) pagination jumps directly to the right position using indexes.

-- OFFSET approach (bad at scale) SELECT ... FROM engagement ORDER BY due_date LIMIT 1000 OFFSET 50000 -- DB scans and discards 50,000 rows every time -- Keyset cursor (good) SELECT ... FROM engagement WHERE due_date > '2025-06-14' OR (due_date = '2025-06-14' AND id > 4821) ORDER BY due_date ASC, id ASC LIMIT 1000 -- Jumps directly to the right position using indexes
Cursor encoding: The cursor is a base64-encoded JSON object {"id": 4821, "val": "2025-06-14"} passed as the ?after= query parameter. It encodes the sort key value and row ID of the last row seen. The cursor resets on any sort change or when filters change significantly.

The tie-breaking id condition ensures stable pagination when multiple rows share the same sort key value — without it, rows at sort key boundaries could be skipped or duplicated.

Filtering

Server-Side Filters

All filter values are passed as parameterized query parameters, never interpolated into SQL strings. Column names are validated against a server-side whitelist before being included in the WHERE clause.

-- Example: office + tax_year filter with cursor SELECT id, engagement_name, client_name, office, tax_year, due_date FROM engagement WHERE office = $1 AND tax_year = $2 AND id > $3 ORDER BY id ASC LIMIT $4 -- All filter values are parameterized — never interpolated into SQL

SQL injection prevention: Column names are validated against a VALID_COLS Set on the server before being used in any query. Filter values are always passed as positional parameters ($1, $2, ...) — never string-concatenated into SQL.
Bandwidth

Dynamic Column Selection

The production engagement schema has 100+ columns, including large TEXT fields that store JSON arrays for efilers, reviewers, preparers, and other staff lists. Transmitting all columns when the user has 15 visible wastes 5–10x bandwidth.

The client sends a ?columns= parameter listing only the columns currently shown. The server validates each column name against the whitelist and constructs a targeted SELECT.

Client: ?columns=engagement_name,client_name,office,due_date,status_id Server: validates each against VALID_COLS Set SELECT id, engagement_name, client_name, office, due_date, status_id FROM engagement ... -- id always included (needed for cursor and row identity)

The column picker sidebar in the UI lets users show or hide any column. Changes trigger a new stream request with the updated column list — the cursor is preserved so the user stays at the same scroll position.

Tree View

Tree View

The tree is built entirely client-side from the flat row stream — same technique as V1, now running on the real production schema. No separate queries are issued per client or entity.

Tree build process
⚙️
Server
Streams flat rows ordered by client_id, entity_id, id. One row per engagement — no nesting, no duplication.
↓
💻
Client — progressive grouping
As rows arrive, client groups by client_id → entity_id. Top of list populates immediately. Expand/collapse state held in Sets — O(1) lookup.

Four view modes are supported: Flat (all rows, no grouping), Client (grouped by client only), Entity (grouped by entity only), and Client + Entity (full two-level hierarchy).

Contrast with production: The current production UI issues a separate query for each client node when the user expands it — the classic N+1 pattern V1 demonstrated. The V2 tree view eliminates this entirely. All grouping happens in the browser from a single flat data stream.
Indexes

Index Strategy

With ~1M rows, the right indexes are the difference between a 5ms index scan and an 800ms heap sort. All indexes below are implemented in the POC and should be mirrored in production.

Single-column indexes — equality filters

Cover WHERE clauses on individual filter fields. PostgreSQL can combine these with bitmap index scans when multiple filters are active.

Implemented
idx_eng_status_id  ·  idx_eng_office  ·  idx_eng_tax_year
idx_eng_svc_type  ·  idx_eng_filing_type  ·  idx_eng_is_active
idx_eng_client_id  ·  idx_eng_entity_id  ·  idx_eng_entity_name
idx_eng_msd (milestone_start_date)
Composite indexes — filter + sort + cursor

The most important indexes for this query pattern. When a user filters by status_id and sorts by due_date, PostgreSQL can satisfy the entire query — WHERE, ORDER BY, and the keyset cursor tiebreak — from a single index scan without touching the heap.

Pattern: (filter_col, due_date NULLS LAST, id)
idx_eng_due_date_id   →  (due_date NULLS LAST, id)
idx_eng_status_due   →  (status_id, due_date NULLS LAST, id)
idx_eng_office_due   →  (office, due_date NULLS LAST, id)
idx_eng_svctype_due  →  (service_type, due_date NULLS LAST, id)
idx_eng_taxyear_due  →  (tax_year, due_date NULLS LAST, id)
idx_eng_filing_due   →  (filing_type, due_date NULLS LAST, id)

The trailing id column is required for the keyset cursor tiebreak — without it, rows with identical due_date values produce ambiguous pagination.

Index-defeating patterns to avoid
PatternWhy it breaks the index
WHERE YEAR(due_date) = 2024Function on indexed column — full scan
WHERE office LIKE '%Vancouver%'Leading wildcard — can't use B-tree
WHERE status_id != 3Negation — index rarely helps

Query Pattern

Stored Procedures vs Dynamic SQL

The production team uses stored procedures. For this query pattern, stored procedures do not add value — and introduce maintenance overhead.

Why stored procedures don't help here

The primary benefit of stored procedures is cached execution plans. That benefit disappears when the procedure uses dynamic SQL internally — which it must, because column lists, filter sets, sort columns, and cursor conditions all vary per request.

Stored proc with dynamic SQL — no plan cache benefit
CREATE PROCEDURE get_engagements(...) AS
  EXECUTE 'SELECT ' + @cols + ' FROM engagement WHERE ' + @where + ' ORDER BY ' + @sort
-- Same dynamic SQL, now wrapped in a procedure layer
-- Plan cache: none (EXECUTE always recompiles)
When stored procedures are worth it
ScenarioWorth it?
Static query called millions of timesYes — plan cache benefit is real
Multi-step atomic transactionYes — single round trip, atomicity
Shared business logic across multiple appsYes — single source of truth
Security isolation (no direct table access)Yes — EXECUTE permission only
Dynamic columns + filters + cursor paginationNo — dynamic SQL negates plan caching
Azure SQL note

SQL Server's plan cache is more sensitive to query variation than PostgreSQL. On Azure SQL, wrapping dynamic SQL in sp_executesql with proper parameterisation does improve plan reuse compared to raw ad-hoc strings — but this is an argument for parameterised dynamic SQL, not for stored procedure wrappers specifically.

Release Notes — 2026-04-23

Dynamic Grouping + Server-Side Facet Bar

Two features added in response to team feedback on POC stability and usability.

1 — Dynamic Two-Tier Grouping

The original tree view had four hardcoded modes (Flat, Client, Entity, Client + Entity). These are replaced by two dropdowns that let users compose any two-level hierarchy from a defined set of fields.

TierAvailable fields
Tier 1 (primary group)Client Group, Entity, Engagement
Tier 2 (sub-group)Entity, Engagement, Due Date, Milestone

The grouping logic is fully generic. buildTree(rows) accepts any two field names and builds the two-level map in a single pass. buildVisibleRows() walks the resulting tree to produce the virtual scroll list, inserting tier1/tier2 header rows at group boundaries. Expand/collapse state is tracked in two Sets keyed by group value — O(1) toggle.

How grouping columns are fetched
💻
Column injection
On every batch fetch, the active tier1 and tier2 field names are appended to the ?columns= parameter regardless of what the user has selected in the column picker. Grouping always has the data it needs — no column visibility requirement.
↓
💻
Group change
When the user changes a grouping dropdown, expand state is cleared and the tree is rebuilt from already-loaded rows. If the new grouping field isn't yet in the loaded data (not previously fetched), the client store is cleared and a fresh batch fetch is triggered.
Known limitation — incomplete groups: Rows are fetched in id order, not pre-sorted by group. A client with 200 engagements spread across the database may have only 12 visible after the first 1,000 rows load — the other 188 arrive in later batches. Both the group header count and the children list are incomplete until all matching rows are loaded. This affects group counts AND the rows shown under each group header. For accurate counts, use the server-side facet bar. For complete child lists, the correct solution is a two-phase approach: fetch group keys server-side first, then lazy-load rows per group on expand.
2 — Server-Side Milestone Facet Bar

Addresses the team feedback that frontend grouping produces unstable counts as data loads in batches. The facet bar runs a dedicated GROUP BY query server-side against the full dataset, returning accurate counts immediately — independent of how many rows have been paginated to the client.

-- GET /api/engagements/groups?groupBy=internal_milestone&office=Toronto SELECT internal_milestone AS value, COUNT(*)::int AS count FROM engagement WHERE office = $1 GROUP BY internal_milestone ORDER BY count DESC -- Returns: [{"value":"In Progress","count":41823}, ...]
Facet bar interaction flow
⚙️
On page load / filter change
Client calls /api/engagements/groups?groupBy=internal_milestone with any active filters. Server runs a single aggregate query and returns all milestone buckets with accurate counts from the full dataset.
↓
💻
Facet bar renders
Milestone pills appear above the filter bar. Each pill shows the milestone name and count. Null values (no milestone assigned) are excluded from the bar.
↓
💻
User clicks a pill
Client resets the row store and refetches with internal_milestone=$value added to the WHERE clause. The clicked pill highlights as active. Clicking again deselects and removes the filter. The facet bar also refetches its counts to reflect the new filter state.

Shared filter builder: Both /api/engagements/stream and /api/engagements/groups use the same buildFilters(query) helper on the server. This guarantees that the counts in the facet bar always reflect the same filter state as the row stream — they can't drift out of sync.
Generalised endpoint: /api/engagements/groups accepts any column in a server-side whitelist (ALLOWED_GROUP_COLS) as the groupBy parameter. The UI currently uses it for internal_milestone, but it could power additional facet bars (by office, service type, status, etc.) with no server changes.
