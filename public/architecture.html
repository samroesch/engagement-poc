The Problem

What is the N+1 query problem?

It happens when you load a list of N items, then make one additional query for each item to load related data. With a second level of nesting (clients → entities → deliverables), it becomes N×M+1.

1
clients
+
200
entities
+
1,000
deliverables
=
1,201 queries
per page load

This pattern often starts as clean-looking code — each layer fetches what it needs. It's only when you multiply by real data volumes that the cost becomes visible.

// Looks clean. Hides the problem. const clients = await getClients() // 1 query for (const client of clients) { client.entities = await getEntities(client.id) // +1 per client = 200 queries for (const entity of client.entities) { entity.deliverables = await getDeliverables(entity.id) // +1 per entity = 1,000 queries } }
Why is this so slow in practice? Even if each query takes only 2ms in the database, each one is a separate round-trip through the network, ORM, connection pool, and API layer. At 2 seconds per call with a concurrency limit of 6, 1,201 queries becomes 201 batches — over 6 minutes of wall-clock time before the user sees data.

The rendering problem compounds the query problem. Once data finally arrives, the browser must create and layout 10,000 DOM nodes — each with cells, text, badges, and event listeners. On a mid-range device this alone adds several seconds and makes scrolling janky.

Where the Performance Comes From

Three independent optimizations

🔗
1. Query Consolidation
Move the join into the database where it belongs. SQL was designed to traverse relationships across tables in a single pass using indexes. Doing it in application code with a loop is reconstructing a JOIN, badly.
1,201 round-trips → 1 query
DB indexes on client_id, entity_id make the join O(n log n)
🪟
2. Virtual Rendering
The browser only needs to display what fits in the viewport — roughly 20 rows. Virtual scroll keeps a flat array of all rows in memory, but only creates DOM nodes for the visible slice. The scrollbar is simulated with a spacer div.
10,000 DOM nodes → ~20 DOM nodes
Constant memory regardless of dataset size
⚡
3. WebSocket Deltas
Instead of polling or reloading to get live status updates, a persistent WebSocket connection lets the server push only what changed. A single status change is a ~60-byte message, not a 10,000-row reload.
Full reload on change → ~60 byte delta
UI stays live without any polling overhead
The Query

What the flat query looks like

The entire data payload — all 10,000 deliverables with their client and entity context — is returned by a single SQL statement. The database uses foreign key indexes to execute this in milliseconds.

-- One query replaces 1,201 SELECT d.id, d.type, d.period, d.status, d.due_date, d.assigned_to, e.id AS entity_id, e.name AS entity_name, e.entity_type, c.id AS client_id, c.name AS client_name, c.industry, c.region FROM deliverables d JOIN entities e ON d.entity_id = e.id -- idx_del_entity JOIN clients c ON e.client_id = c.id -- idx_ent_client ORDER BY c.name, e.name, d.type

The result is a flat list of denormalized rows — each row carries its own client and entity context. The browser reassembles the tree client-side using a two-pass Map lookup, which is O(n) and takes about 1ms for 10,000 rows.

// Client-side tree build — O(n), one pass const clientMap = new Map(), entityMap = new Map() for (const row of flatRows) { if (!clientMap.has(row.client_id)) clientMap.set(row.client_id, { ...clientFields(row), entities: [] }) if (!entityMap.has(row.entity_id)) { const e = { ...entityFields(row), deliverables: [] } entityMap.set(row.entity_id, e) clientMap.get(row.client_id).entities.push(e) } entityMap.get(row.entity_id).deliverables.push(deliverableFields(row)) }
Sorting

How column sort triggers a new stream

Sorting in the flat view can't be done cheaply in the browser — the dataset is 100,000 rows and the browser only holds what has been streamed so far. Instead, clicking a column header fires a new server-sent event stream with the requested sort order, and the server re-runs the query with a different ORDER BY. The table repopulates as the first batch of rows arrives — exactly like the initial load.

Tree view sorts differently. In the grouped tree view (Client, Entity, or Client + Entity), all 100,000 deliverables are already in memory from the initial load. Clicking a column header re-sorts the in-memory array client-side and rebuilds the visible rows — no server round-trip at all. Sort + re-render takes ~2ms for 100,000 rows.

The distinction matters: flat view needs server-side sort because it can't hold the full dataset in sorted order client-side without re-querying. Tree view already has everything in memory, so it can sort for free. Both use the same column headers and sort state — the difference is invisible to the user.

Virtual Rendering

How virtual scroll works

A virtual list maintains two things: a flat array of all rows (in memory, not DOM), and a viewport that only renders the slice currently visible. As the user scrolls, rows are recycled — the same ~20 DOM nodes are repositioned and their content is swapped.

Virtual scroll mechanics
Memory
visibleRows[] — 1,200 items
Each item: plain JS object, no DOM
Filters/group changes rebuild this array
Expand/collapse changes this array
~300KB RAM for 10,000 rows
DOM
Spacer div: height = rows × 42px
Row container: transform: translateY
Only renders scroll window ± 8 rows
On scroll: recalculate slice, swap content
~20 DOM nodes regardless of dataset size
// Virtual render — called on every scroll event function renderRows() { const scrollTop = viewport.scrollTop const viewH = viewport.clientHeight const totalRows = visibleRows.length // Which rows are in the viewport? const startIdx = Math.max(0, Math.floor(scrollTop / ROW_H) - OVERSCAN) const endIdx = Math.min(totalRows, Math.ceil((scrollTop + viewH) / ROW_H) + OVERSCAN) spacer.style.height = (totalRows * ROW_H) + 'px' // fake total height rowContainer.style.transform = `translateY(${startIdx * ROW_H}px)` // position visible slice rowContainer.innerHTML = visibleRows .slice(startIdx, endIdx) .map(row => renderRow(row)) // only render what's visible .join('') }

Production Readiness

Indexes and join performance at scale

Joins are not inherently slow. A JOIN across several tables on properly indexed columns can return results in milliseconds even at millions of rows. The danger is that joins amplify any indexing mistake — a missing index that costs 10ms at 10,000 rows costs 10 seconds at 10 million.

The rule: every join column must be indexed

When the database executes a JOIN, it needs to look up matching rows in the joined table. Without an index, it scans the entire table for every row on the left side — an O(n²) operation. With an index, it's a B-tree lookup: O(log n) per row.

-- For every JOIN like this: JOIN entities e ON d.entity_id = e.id JOIN clients c ON e.client_id = c.id JOIN staff s ON d.assigned_to = s.id JOIN service_lines sl ON d.service_line_id = sl.id -- You need an index on the FK side of each join: CREATE INDEX idx_del_entity ON deliverables(entity_id) CREATE INDEX idx_del_assigned ON deliverables(assigned_to) CREATE INDEX idx_del_service_line ON deliverables(service_line_id) CREATE INDEX idx_ent_client ON entities(client_id) -- The PK side (e.id, c.id, s.id) is already indexed by default -- It's the FK side living in the large table that's usually missing

The pattern to check: for every JOIN … ON big_table.some_id = other_table.id, there should be an index on big_table.some_id. The other_table.id side is a primary key and already indexed. It's always the foreign key column in the large table that gets missed.

Index filter columns too — not just join columns

Join indexes get you a fast join. But if your query also filters (WHERE office = 'xyz' AND due_date > '2025-01-01'), the database still has to evaluate those conditions. Without indexes on filter columns, it scans all joined rows to find the matching ones.

-- Common filter columns that should be indexed: CREATE INDEX idx_del_status ON deliverables(status) CREATE INDEX idx_del_due_date ON deliverables(due_date) CREATE INDEX idx_del_period ON deliverables(period) CREATE INDEX idx_cli_office ON clients(office) CREATE INDEX idx_cli_region ON clients(region) -- For common filter combinations, a composite index can be even faster: CREATE INDEX idx_del_office_date ON deliverables(office, due_date) -- ↑ covers WHERE office = ? AND due_date BETWEEN ? AND ? in one index scan
Sort columns need indexes too. An ORDER BY due_date without an index forces the database to sort the entire result set in memory or on disk after the JOIN. At 10M rows this can add seconds. An index on the sort column lets the DB read rows in already-sorted order — no sort step needed at all.

Pitfalls that silently break indexes

Having an index isn't enough if the query is written in a way that prevents the DB from using it. These are the most common ways a valid index gets bypassed:

Index-defeating patterns to avoid
1. Functions on indexed columns
WHERE YEAR(due_date) = 2025 -- function prevents index use WHERE due_date BETWEEN '2025-01-01' AND '2025-12-31' -- index used ✓ WHERE LOWER(client_name) = 'acme' -- function prevents index use WHERE client_name = 'Acme' -- index used ✓ (store normalised)
Any function wrapping a column forces a full scan. The index is on the raw value, not the computed result.
2. Implicit type coercion
WHERE client_id = '42' -- client_id is INT, '42' is VARCHAR → full scan WHERE client_id = 42 -- types match → index used ✓
When column type and parameter type don't match, the DB coerces every row to compare — defeating the index. Common when IDs come through an ORM as strings.
3. Leading wildcard LIKE
WHERE client_name LIKE '%corp' -- leading wildcard → full scan WHERE client_name LIKE 'corp%' -- trailing wildcard → index used ✓
B-tree indexes work left-to-right. A leading wildcard means "could be anything at the start" — the index can't help. If you need full-text search, use a dedicated FTS index.
4. OR across different columns
WHERE status = 'filed' OR due_date < '2025-01-01' -- DB often can't use both indexes simultaneously → may fall back to full scan SELECT * FROM deliverables WHERE status = 'filed' UNION ALL SELECT * FROM deliverables WHERE due_date < '2025-01-01' AND status != 'filed' -- each branch uses its own index ✓
OR conditions across different columns are hard for query planners. UNION ALL lets each branch use its own index independently.
5. Stale statistics
-- Postgres: refresh planner statistics after bulk loads ANALYZE deliverables -- SQL Server equivalent UPDATE STATISTICS deliverables
The query planner decides whether to use an index based on estimated row counts. After bulk inserts or deletes, statistics can be stale — causing the planner to choose a full scan even when an index would be faster. Most DBs auto-analyze on a schedule, but after a large data migration, run it manually.

How to diagnose: EXPLAIN

Every major database has an explain plan command that shows exactly how a query will execute — which indexes it uses, which it ignores, and where the cost is. This is the first tool to reach for when a query is slow.

-- Postgres: EXPLAIN ANALYZE actually runs the query and shows real timings EXPLAIN ANALYZE SELECT d.*, e.name, c.name FROM deliverables d JOIN entities e ON d.entity_id = e.id JOIN clients c ON e.client_id = c.id WHERE c.office = 'London' AND d.due_date > '2025-01-01' -- Look for these warning signs in the output: -- Seq Scan on deliverables ← full table scan, no index used -- rows=100000 (actual rows=12) ← planner estimate was way off (stale stats) -- Hash Join ← can be fine, but Nested Loop + Index Scan is usually faster for FK joins -- Sort (cost=...) ← in-memory sort, ORDER BY column probably not indexed

The goal is to see Index Scan or Index Only Scan on your large tables, not Seq Scan. If you see a seq scan on a table with millions of rows, that's where to add the missing index.

Future Thinking — Not Implemented

Progressive loading at scale

The current POC loads all 100,000 deliverables on every page open. For a real dataset in the hundreds of thousands or millions, that becomes impractical — even with streaming, you're transferring megabytes the user may never scroll to. This section captures the thinking on how to fix that without sacrificing the filter-sort-scroll experience.

Nothing below is implemented. This is forward-looking design thinking only. The live demo uses full-load streaming for simplicity and to keep the POC focused on the N+1 vs flat query comparison.

Why full load breaks at scale

With 100,000 rows the current approach works fine — the JOIN query returns in <50ms and streaming delivers the first batch in ~100ms. At 1 million rows, that first query takes longer, the stream takes ~10 seconds to complete, and the browser is holding ~30MB of JavaScript objects in memory. At 10 million rows it's simply not viable.

The obvious fix — "just paginate" — creates a new problem: filters and sorts apply across the full dataset, not just the loaded page. A user filtering to a specific status might need to see rows that are on page 47. You can't filter client-side data you haven't loaded.

The right first step: server-side scoping

Before adding pagination complexity, the most impactful change is reducing the dataset at source. In a real engagement tool, users always operate within a context — a portfolio, a team, a date range. Scoping the query server-side to "deliverables for this user's assigned clients, due in the next 90 days" might take a dataset from 1 million rows down to 5,000 before any pagination is needed. Most users never need all the data.

Server-scoped query — scope first, then stream

Filter-triggered continuation loading

For cases where even the scoped dataset is large, a hybrid approach lets you load progressively without losing filter coverage. The key insight: when a filter knocks out most of the loaded rows, the user is implicitly telling you they want to see further into the dataset.

// Client state for progressive loading let loadedOffset = 0 // how many rows the server has sent so far let hasMore = true // server sets this false when cursor is exhausted const LOAD_SIZE = 1000 // rows per fetch const THRESHOLD = 200 // if filtered rows drop below this, fetch more function onFilterChange() { applyFilters() // client-side, instant if (filteredRows.length < THRESHOLD && hasMore) { fetchMore(loadedOffset) // filtered result is thin — pull next batch } } async function fetchMore(offset) { // ?offset=N tells the server where the cursor left off const res = await fetch(`/api/flat-stream?sort=${sortCol}&dir=${sortDir}&offset=${offset}`) // Server streams next LOAD_SIZE rows, then sends: {"type":"meta","hasMore":true,"offset":2000} // Merge into deliverableMap, re-apply filters, update loadedOffset + hasMore }

This gives you snappy client-side filtering on loaded data, with automatic backfill when the filter result is sparse. The user experience is seamless — apply a filter, see results immediately from loaded data, and watch more appear if the dataset continues.

Why sort changes require a full reset

Continuation loading has one hard constraint: it only works within a fixed sort order. If you've loaded rows 0–1000 sorted by due date, and the user clicks "sort by assigned to", the rows you have in memory are no longer in the right relative order. You must start a fresh stream from offset 0 with the new ORDER BY.

This is actually fine — sort is already a server round-trip in flat mode, so the UX expectation is already set. The pagination cursor just resets as part of the sort change.

Keyset pagination vs OFFSET. For deep continuation fetches, OFFSET 50000 forces the database to scan and discard 50,000 rows every time. Keyset pagination (cursor-based) avoids this: instead of an offset number, you pass the last-seen value of the sort column, and the query uses WHERE due_date > '2025-06-14' to start exactly where you left off. This keeps continuation fetches fast regardless of how deep into the dataset you are — but it requires the sort column to be indexed and the cursor value to be stable between fetches.
